  9. Images, Objects and Applets  
  9.1 Introduction  
  HTML presents multimedia features which allow you to include images, applets (programs that are automatically downloaded and run on the user's machine), video clips, and other HTML documents in their pages.

Versions of HTML prior to version 4 allowed authors to include images via the img element and applets via the applet element. These elements have several limitations:

  • They fail to solve the more general problem of how to include new and future media types.
  • The applet element only works with Java-based applets. This element is deprecated in favor of the object element.
  • They pose accessibility problems.

However, HTML 4.0 introduced the object element, which offers an all-purpose solution to generic object inclusion. The object element allows HTML authors to specify everything required by an object for its presentation by a user agent: source code, initial values, and run-time data. In this specification, the term "object" is used to describe the things that people want to place in HTML documents; other commonly used terms for these things are: applets, plug-ins, media handlers, etc.

To include images, you may use the object element or the img element.

To include applets, you should use the object element as the applet element is deprecated.

To include one HTML document in another, you may use either the new iframe element or the object element. In both cases, the embedded document remains independent of the main document. Visual user agents may present the embedded document in a distinct window within the main document.

Images and other included objects may have hyperlinks associated with them, both through the standard linking mechanisms, but also via image maps.

Note that although the object element is intended to replace the more specific img and applet elements, as well as the proprietary embed and bgsound elements, a lack of browser support and severe bugs in supporting browsers make the other elements a better choice for now.

  
     
  9.2 The img Element  
 

Syntax    <img>

List of img Attributes

The img element embeds an image in the current document at the location of the element's declaration. The img element has no content; it is usually replaced inline by the image designated by the src attribute, the exception being for left or right-aligned images.

Eg.1 We can insert an image directly into a document as follows:

 
<body>
<p>Here is a photo of our product range:
<img src="http://www.mycompany.com/images/products.jpg"
     width="70" height="70" alt="A photo of our product range."></p>
</body>

Eg.2 The image can also be inserted using the object element as follows:

 
<body>
<p>Here is a photo of our product range:
<object data="http://www.mycompany.com/images/products.jpg"
        type="image/jpg" width="70" height="70">
A photo of our product range.
</object></p>
</body>

The alt attribute specifies alternate text that is displayed when the image cannot be rendered. User agents display alternate text when they cannot render images, cannot support a certain type of image or when users configure them not to display images.

  
     
  9.3 The object Element  
 

Syntax    <object>...</object>

List of object Attributes

Most user agents have built-in mechanisms for rendering common data types such as text, images, colors, fonts, and graphic elements. To render data types they don't support themselves, user agents generally run external applications. The object element allows you to control whether data should be rendered externally or by some program which is specified by you, that renders the data within the user agent.

In most cases, three things need to be specified:

  • The implementation of the included object. For instance, if the included object is a navigation applet, you must indicate the location of the applet's executable code.
  • The data to be rendered. For instance, if the included object is a program that renders font data, you must indicate the location of that data.
  • Additional values required by the object at run-time. For example, some applets may require initial values for parameters.

The object element allows you to specify all three types of data. However, all three types of data may not need to be specified at once. For example, some objects may not require data. Others may not require run-time initialization. Others may not require additional implementation information, i.e., the user agent itself may already know how to render that type of data, for example, GIF images.

You use the object element to specify an object's implementation and the location of the data to be rendered. However, to specify run-time values, you use the param element.

In the following example, we insert an analog clock applet, complete with alternate text, in a document via the object element. The applet, written in the Python language (hence .py extension), requires no additional data or run-time values. The classid attribute specifies the location of the applet:

 
<p><object classid="http://www.somesite.com/analogclock.py">
An animated clock.
</object></p>

One significant consequence of the object element is that it offers a mechanism for specifying alternate object renderings. Each embedded object declaration may specify alternate content types. If a user agent cannot render the outermost object, it tries to render the contents, which may be another object element, etc.

In the following example, several object declarations are embedded to illustrate how alternate renderings work. A user agent will attempt to render the first object element it can, in the following order: (1) a solar-eclipse applet written in the Python language, (2) an MPEG animation of the eclipse, (3) a GIF image of the eclipse, (4) alternate text.

 
<p>              <!-- First try the applet -->               
<object title="The solar eclipse as seen from the observatory" 
        classid="http://www.somesite.com/eclipse.py">
                <!-- If unable to, try the MPEG video -->
   <object data="eclipse.mpeg" type="application/mpeg">
                 <!-- If unable to, try the GIF image --> 
    <object data="eclipse.gif" type="image/gif">
                 <!-- Otherwise use the last resort and display the text -->
      The solar eclipse as seen from the observatory.
    </object>
  </object>
 </object></p>

The outermost declaration specifies an applet that requires no data or initial values. The second declaration specifies an MPEG animation and, since it does not define the location of an implementation to handle MPEG, relies on the user agent to handle the animation. The type attribute is also set so that a user agent that knows it cannot render MPEG will not bother to retrieve "eclipse.mpeg" from the network. The third declaration specifies the location of a GIF image and furnishes alternate text in case all other mechanisms fail.

  
     
  9.4 Object initialization: the param Element  
 

Syntax    <param>

List of param Attributes

param elements specify a set of values that may be required by an object at run-time. Any number of param elements may appear in the content of an object or applet element, in any order, but must be placed at the start of the content of the enclosing object or applet element.

To illustrate the use of the param element, let's return to the clock example given above. If the applet is able to handle two run-time parameters that define its initial height and width, we can set the initial dimensions to 60x60 pixels with two param elements.

 
<p><object classid="http://www.somesite.com/analogclock.py">
<param name="height" value="60" valuetype="data">
<param name="width" value="60" valuetype="data">
Your user agent cannot render this application.
</object></p>

In the following example, run-time data for the object's "location" parameter is specified as an external resource (a GIF file). The value of the valuetype attribute is thus set to "ref" and the value is a URI designating the resource.

 
<p><object classid="http://www.somesite.com"
       standby="Loading photo, please wait...">
<param name="location" value="/images/product.gif" valuetype="ref">
</object></p>

A standby attribute has also been set so that the user agent may display a message while the rendering mechanism loads.

  
     
  9.5 Naming Schemes For Objects  
  The location of an object's implementation is given by a URI. The first segment of an absolute URI specifies the naming scheme used to transfer the data designated by the URI. For HTML documents, this scheme is frequently "http". Some applets might use other naming schemes. For instance, when specifying a Java applet, authors may use URIs that begin with "java" and for ActiveX applets, authors may use "clsid".

In the following example, we insert a Java applet into an HTML document.

 
<p><object codetype="application/java" classid="java:app.start"
codebase="http://www.somesite.com/java/archive/">
</object></p>

By setting the codetype attribute above, a user agent can decide whether to retrieve the Java application based on whether it supports the application or not.

Some rendering schemes require additional information to identify their implementation and must be told where to find that information. Path information may be given to the object's implementation via the codebase attribute.

  
     
  9.6 The applet Element  
 

Syntax    <applet>...</applet>

List of applet Attributes

Applet is deprecated together with all of its attributes in favor of the object element.

The applet element, supported by all Java-enabled browsers, allows designers to embed a Java applet in an HTML document.

The content of the applet acts as alternate information for user agents that don't support this element or that have been configured not to support applets. User agents must ignore the content otherwise.

DEPRECATED EXAMPLE USING THE APPLET ELEMENT:
In the following example, the applet element includes a Java applet in the document. Since the applet is in the same directory as the current document, no codebase is supplied.

 
<p><applet code="clock.class" width="500" height="500">
Java applet of an animated clock.
</applet></p>

The above example may be re-written using the object element as follows:

 
<p><object codetype="application/java" classid="java:clock.class" width="500" height="500">
Java applet of an animated clock.
</object></p>

Initial values may be supplied to the applet via the param element.

DEPRECATED EXAMPLE USING THE APPLET ELEMENT:
Audio playing java applet:

 
<p><applet code="Audioapp" width="15" height="15">
<param name="audible" value="sounds.wav">
Audio playing java applet.
</applet></p>

The above can be re-written using the object element as follows:

 
<p><object codetype="application/java" classid="Audioapp" width="15" height="15">
<param name="audible" value="sounds.wav">
Audio playing java applet.

</object></p>

  
     
  9.7 Embedded Documents  
  Rather than linking to a document, it is possible to embed the document directly into an HTML document. To do this one can use either the iframe element or the object element for this purpose, but the elements differ in some respects. Apart from the two elements having different content models, the iframe element may be a target frame and may be "selected" by a user agent as the focus for printing, etc. User agents may render selected frames elements in ways that distinguish them from unselected frames, such as by drawing a border around the selected frame.

An embedded document is entirely independent of the document in which it is embedded. For instance, relative URIs within the embedded document resolve according to the base URI of the embedded document, not that of the main document. An embedded document is only rendered within another document, for example, in a subwindow. It remains otherwise independent.

For example, the following code embeds the contents of included_doc.html at the location where the object definition occurs.

 
...previous text...
<object data="included_doc.html">
The document included_doc.html could not be embedded.
</object>
...text after...

The contents of the object element must only be rendered if the file specified by the data attribute cannot be loaded.

  
     
  9.8 Image Maps  
  Using image maps, it is possible to specify regions of an image or object and assign a specific action to each region, such as to retrieve a document or run a program, etc. When the region is activated by the user, e.g. by clicking on it, the action is executed.

An image map is created using a list of co-ordinates that define the mapping regions and which URI's those co-ordinates are associated with. This is known as the map file. To create an image map you should use an image mapping program or a graphics program capable of image mapping.

There are two types of image maps:

  • Client-side. When a user activates a region of a client-side image map with a mouse, the pixel coordinates are interpreted by the user agent. The user agent selects a link that was specified for the activated region and follows it.
  • Server-side. When a user activates a region of a server-side image map with a mouse, the pixel coordinates of the click are sent to the server-side agent specified by the href attribute of the a element. The server-side agent interprets the coordinates and performs some action.

Client-side image maps are preferred over server-side image maps for at least two reasons:

  • they are accessible to people browsing with non-graphical user agents and

they offer immediate feedback as to whether or not the pointer is over an active region.

 

  
     
  9.9 The map and area Elements  
 

Syntax    <map>...</map>

List of map Attributes

Syntax    <area>

List of area Attributes

The map element specifies a client-side image map that may be associated with other elements, such as img, object, or input. An image map is associated with an element via the element's usemap attribute.

The presence of the usemap attribute for an object implies that the object being included is an image. Furthermore, when the object element has an associated client-side image map, user agents may implement user interaction with the object solely in terms of the client-side image map. This allows user agents, such as an audio browser, to interact with the object without having to process it. The user agent may even choose not to retrieve or process the object. When an object has an associated image map, it should not be expected that the object will be retrieved or processed by every user agent.

The map element content model allows the following to be combined:

  1. One or more area elements. These elements have no content but specify the geometric regions of the image map and the link associated with each region. User agents do not generally render area elements, therefore alternate text must be provided for each area with the alt attribute.
  2. Block-level content. This content should include a elements that specify the geometric regions of the image map and the link associated with each region. User agents should render block-level content of a map element, therefore this method should be used to create more accessible documents.

Note that when a map element contains both area elements and block-level content, user agents must ignore the area elements.

An image maps's geometry should be specified completely with area elements, or completely with a elements, or completely with both if content is mixed. Mixing content enables older user agents to handle map geometries specified by area elements and new user agents to take advantage of richer block content.

Client-side image map examples

In the following example, a client-side image map is created for the object element. In this example we do not want to render the image map's contents when the object is rendered, so we "hide" the map element within the object element's content. Consequently, the map element's contents will only be rendered if the object cannot be rendered.

 
   <body>
     <p><object data="imagemap.gif" type="image/gif" usemap="#examplemap">
     <map name="examplemap">
       <p>Navigation links:
       <a href="products.html" shape="rect" coords="0,0,75,73">Our Products</a> 
       <a href="prices.html" shape="rect" coords="75,0,148,73">Our Prices</a> 
       <a href="specials.html" shape="rect" coords="0,73,75,147">Special Offers</a> 
       <a href="search.html" shape="rect" coords="75,73,148,147">Search</a>
      </p></map>
     </object></p>
   </body>

For instance, we may want to associate an image map with an object element and include a text navigation bar at the bottom of the page. To do so, we define the map element outside of the object element:

  9.10 Server-side Image Maps  
  Server-side image maps may be used in situations where the image map is too complicated for a client-side image map.

It is only possible to define a server-side image map for the img and input elements. In the case of the img element, the img element must be inside an a element and the boolean attribute ismap must be set. In the case of the input element, the input must be of type "image".

When the user activates the link by clicking on the image, the screen coordinates are sent directly to the server where the document resides. Screen coordinates are expressed as screen pixel values relative to the image.

In the example below, the active region defines a server-side link. Therefore, clicking anywhere on the image will cause the click's coordinates to be sent to the perl script located in the cgi-bin directory on the server.

 
<p><a href="http://www.somesite.com/cgi-bin/bannerads">
        <img src="banner1.gif" ismap alt="banner1"></a></p>

The location clicked is passed to the perl script on the server as follows. The user agent derives a new URI from the URI specified by the href attribute of the a element, by appending '?' followed by the x and y coordinates, separated by a comma. The link is then followed using the new URI. For instance, in the given example, if the user clicks at the location x=18, y=35 then the derived URI
is "http://www.somesite.com/cgi-bin/bannerads?18,35".

 

  
     
  9.11 Dimensions & Presentation of Images, Objects, and Applets  
  All img and object attributes that concern visual alignment and presentation have been deprecated in favor of style sheets.

List of img Attributes

List of object Attributes

List of applet Attributes

Width and height

width = pixels or percentage - Image and object width override.

height = pixels or percentage - Image and object height override.

When specified, the width and height attributes tell user agents to override the natural image or object size in favor of the stated values. However, specifying actual values for the height and width attributes gives user agents an idea of the size of an image (Eg.1 above) or object (Eg.2 above) so that they may reserve space for it and continue rendering the document while waiting for the image data. This results in the page being rendered faster. Most graphics programs are able to give the width and height dimensions of an image.

White space around images and objects

hspace = pixels - (Deprecated attribute) This attribute specifies the amount of white space to be inserted to the left and right of an image, applet, or object. If no hspace value is specified, the default value is generally a small, non-zero length.

vspace = pixels - (Deprecated attribute) This attribute specifies the amount of white space to be inserted above and below an image, applet, or object. If no vspace value is specified, the default value is generally a small, non-zero length.

Borders

An image or object may be surrounded by a border, for example when a border is specified by you or when the image is the content of an a element.

border = pixels - (Deprecated attribute) This attribute specifies the width of an image's or object's border, in pixels. If no border value is specified, the default value for this attribute depends on the user agent.

Alignment

align = [ top | middle | bottom | left | right ] (Deprecated attribute) This attribute specifies the position of an image, object, or applet with respect to its context (surrounding text).

9.12 Specifying Alternate Text

alt = text - This attribute specifies alternate text for user agents that cannot display images, forms, or applets. The lang attribute specifies the language of the alternate text.

The img, area, applet, and input elements let you specify alternate text to serve as content when the element cannot be rendered normally. Specifying alternate text assists users whose browsers don't support forms, visually impaired users, those who use speech synthesizers and those who have configured their browsers not to display images.

The alt attribute must be specified for the img and area elements. It is optional for the input and applet elements.

While alternate text may be very helpful, do not specify irrelevant or meaningless alternate text as this will annoy users and slow down user agents that must convert text to speech or braille output. When using the alt attribute for img and area elements, rather than using a text string that is irrelevant or meaningless, use alternate text that consists of an empty string, (i.e. alt="").

 

  
     
  9.13 Supported Image Formats  
  GIF: This is an acronym for Graphics Interchange Format. This format uses a maximum of 256 colors and all other colors are created by combining various proportions of these colors. The GIF format is best for displaying images such as icons, buttons and logos that have been created using a graphics program.

GIF 89a is the current version and has the following features:

  1. Interlacing: Saving the GIF 89a image as interlaced will enable the browser to display the image as it loads, getting gradually smoother and clearer until it is finished loading. Non-interlaced GIFs only appear once the whole image has loaded. Interlaced GIFs have slightly larger file sizes than non-interlaced GIFs.
  2. Transparency: You can set a single color to be transparent with a GIF 89a image. This means that it will allow the background color or image to show through it.
  3. Animated GIFs: GIF 89a images can be animated using special GIF animation software. An animated GIF consists of a number of separate GIF images which are saved into a file and then looped once, a certain number of times or indefinitely.

Images in this format are saved with the .gif extension, i.e. image_name.gif

JPEG: This is an acronym for Joint Photographic Experts Group. This format is good for photographs because JPEG files can contain up to 16 million colors. You don't have the options of including transparency or interlacing images with JPEG images. However, JPEG images allow you to specify the degree of file compression, thereby allowing you to decide between a balance of image quality and file size.

There is also a newer version of the JPEG format known as Progressive JPEG. The Pro-JPEG format provides superior compression compared to ordinary JPEGs and also supports interlacing.

Images in this format are saved with the .jpg extension, i.e. image_name.jpg

PNG: This is an acronym for Portable Network Graphics. This format is good for images that have a combination of text and graphics, for example, program GUI screenshots. The PNG format allows variable transparency, fast interlacing and truecolor images.

Images in this format are saved with the .png extension, i.e. image_name.png
