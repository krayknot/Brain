  14. Scripts  
  14.1 Introduction  
  A client-side script is a program that may be embedded directly in an HTML document or may accompany it as an external script document. The program executes on the client's machine when the document loads, or at some other time such as when a drop-down navigation menu is activated. HTML's support for scripts is independent of the scripting language (e.g. Javascript, VBScript, and Tcl). Javascript is the most widely used scripting language followed by VBScript.

Scripts offer you a way to create dynamic Web pages. For example:

  • Scripts can modify the contents of a document dynamically as it loads.
  • Scripts may accompany a form to process input as it is entered. These scripts ensure that the input data conforms to predetermined ranges of values and characters etc.
  • Scripts may be triggered by intrinsic events, such as loading, unloading, element focus, mouse movement, etc.
  • Scripts may be linked to form controls (e.g., buttons) to produce graphical user interface elements.

There are two types of scripts that you can attach to an HTML document:

  • Those that are executed one time when the document is loaded by the user agent. Scripts that appear within a script element are executed when the document is loaded. You can include alternate content via the noscript element for user agents that do not handle scripts.

Those that are executed every time a specific event occurs. These scripts may be assigned to a number of elements via the intrinsic event attributes.

  
     
  14.2 Designing Documents For Script Supporting User Agents  
 

The script element

Syntax    <script>...</script>

List of script Attributes

The script element is used to place a script within a document. This element may appear any number of times in the head or body of an HTML document.

The script may be defined within the contents of the script element or preferably in an external file. If the src attribute is not set, user agents must interpret the contents of the element as the script. If the src has a URI value, user agents must ignore the element's contents and retrieve the script via the URI.

  
     
  14.3 Specifying The Scripting Language  
  HTML does not rely on a specific scripting language, therefore you must explicitly tell user agents the language of each script. This may be done either through a default declaration or a local declaration.

The default scripting language

You should specify the default scripting language for all scripts in a document by including the following meta declaration in the head:

 
     <meta http-equiv="Content-Script-Type" content="type">

where "type" is a content type naming the scripting language, examples being "text/javascript", "text/vbscript", "text/tcl".

Documents that do not specify default scripting language information and that contain elements that specify an intrinsic event script are incorrect. User agents may still attempt to interpret incorrectly specified scripts but are not required to do so.

Local declaration of a scripting language

The type attribute must be specified for each script element instance in a document. The value of the type attribute for a script element overrides the default scripting language for that element.

In this example, we declare the default scripting language to be "text/tcl". We include one script in the header, whose script is located in an external file and is in the scripting language "text/vbscript". We also include one script in the body, which contains its own script written in "text/javascript".

 
<!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.0 Transitional//EN">
<html>
  <head>
    <title>Specifying Scripts</title>
        <meta http-equiv="Content-Script-Type" content="text/tcl">
    <script type="text/vbscript" src="http://www.somesite.com/scripts/vbmode"></script>
   </head>
<body>
<script language="javascript">
...JavaScript contents here...
</script>
</body>
</html>

The local declaration of an external javascript document is as follows:

 
<!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.0 Transitional//EN">
<html>
  <head>
    <title>Specifying Scripts</title>
    <script language="javascript" src="name_of_file.js"></script>
   </head>
<body>
    ...document contents here...
</body>
</html>

In the above example, name_of_file.js is the name of the external script document. In this case it is in the same folder (directory) as the web pages (web page files). Note that external javascript documents are saved with the .js extension. It is in this document (file) that you place all the javascript scripts that you are going to be using on your site, but do not include the opening and closing script tags for any of the individual scripts, i.e. leave out the <script language="javascript"> and </script> tags and insert only what's between them in this document. Name the document and save it with the .js extension in the same directory as your web pages or in a different directory if you wish (in which case be sure to give the correct path to it when stating the src value.

  
     
  14.4 Intrinsic Events  
  List of Intrinsic Events

It is possible to associate an action with the intrinsic events that occur when a user interacts with a user agent. Each of the "intrinsic events" in the popup list (click above link) takes a value that is a script. The script is executed whenever the event occurs for that element.

Control elements such as input, select, button, textarea, and label all respond to certain intrinsic events.

  
     
  14.5 Dynamic Modification Of Documents  
  Scripts that are executed when a document is loaded may be able to modify that document's contents dynamically. The ability to do so depends on the scripting language itself, for example, the "document.write" statement in Javascript.

The following example illustrates how scripts can modify a document dynamically:

 
<script type="text/javascript">
   document.write("<p><b>This is dynamic modification of a document.</b></p>")
</script>

It has the same effect as the following markup:

 

<p><b>This is dynamic modification of a document.</b></p>

  
     
  14.6 Designing Documents For Non-Supporting User Agents  
 

The noscript element

Syntax    <noscript>...</noscript>

The noscript element allows you to provide alternate content when a script is not executed. The content of a noscript element should only be rendered by a script-aware user agent in the following cases:

  • The user agent is configured not to evaluate scripts.
  • The user agent doesn't support a scripting language invoked by a script element earlier in the document.

User agents that do not support client-side scripts must render this element's contents.

In the following example, a user agent that executes the script will include some dynamically created information in the document. If the user agent doesn't support scripts, the user can retrieve the data through a link.

 
<script language="javascript">
 ...script contents to insert information...
</script>
<noscript>
<p>To get the information click <a href="http://www.somesite.com/info">here</a>.</p>

</noscript>
