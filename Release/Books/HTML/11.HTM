  12. Forms  
  12.1 Introduction  
  A form is a section of a document containing special elements called controls (text boxes, password boxes, textareas, checkboxes, radio buttons, menus, buttons, etc.), and labels on those controls. Users usually complete a form by entering text in text boxes, password boxes or textareas, checking boxes or radio buttons, selecting menu items, etc., before submitting the form to a Web or mail server for processing. Web servers normally have to have some sort of script installed on them, usually CGI scripts, to gather and process information from the more elaborate forms. Some forms containing, for example, menus (such as navigation menus), can be processed using javascript.

Eg.1 Below is a simple form that includes labels, text boxes, and push buttons (i.e. the submit and reset buttons):

 
<form name="Creditcard" action="http://www.somesite.com/cgi-bin/userinfo.pl" method="post">
Card Holders Name: <input type="text" name="CustomerName" size="20"><br>
Card Number: <input type="text" name="Number" size="20" maxlength="20"><br>
Expiry Date: <input type="text" name="Month" size="2" maxlength="2" value="MM">
/<input type="text" name="Year" size="2" maxlength="2" value="YY"><br>
<input type="submit" value="Submit"> <input type="reset" value="Reset">

</form>

  
     
  12.2 Form Controls  
  Users interact with forms through named controls. A control's name is given by its name attribute within a form element.

Each control has both an initial value and a current value, both of which are character strings. The control's current value is first set to the initial value. After that, the control's current value may be modified through user interaction and scripts. A control's initial value does not change. Therefore each control's current value is reset to its initial value when a form is reset. If a control does not have an initial value, the effect on that control of resetting the form is undefined.

When a form is submitted for processing, some controls have their name paired with their current value and these pairs are submitted with the form. Those controls for which name/value pairs are submitted are called successful controls.

  
     
  12.3 Control Types  
  The following control types exist:

button

Three types of buttons may be created:

·         submit buttons: When activated, a submit button submits a form. A form may contain more than one submit button.

·         reset buttons: When activated, a reset button resets all controls to their initial values.

·         push buttons: Push buttons have no default behavior. Each push button may have client-side scripts associated with the element's event attributes. When an event occurs (e.g., the user presses the button, releases it, etc.), the associated script is triggered.

The scripting language of a push button script should be specified through a default script declaration.

Buttons are created with the button element or the input element. The button element offers richer rendering capabilities than the input element.

checkbox

Checkboxes (and radio buttons) are on/off switches that may be toggled by the user. A switch is on when the control element's checked attribute is set. When a form is submitted, only "on" checkbox controls can become successful.

Several checkboxes in a form may share the same control name. Therefore, checkboxes allow users to select several values for the same property. The input element is used to create a checkbox control.

radio button

Radio buttons are like checkboxes except that when several share the same control name, they are mutually exclusive. That is, when one is switched "on", all others with the same name are switched "off". The input element is used to create a radio button control.

One of the radio buttons in a set should be checked at all times. If none of the input elements of a set of radio buttons specifies "checked", then the user agent must check the first radio button of the set initially. However, you should ensure that in each set of radio buttons, one is initially "on" since user agents behave differently.

menu

Menus offer users options from which to choose. The select element creates a menu, in combination with the optgroup and option elements.

text input

Two types of controls can be created that allow users to input text. The input element creates a single-line input control and the textarea element creates a multi-line input control. In both cases, the input text becomes the control's current value.

file select

This control type allows the user to select files so that their contents may be submitted with a form. The input element is used to create a file select control.

hidden control

You can create controls that are not rendered but whose values are submitted with a form. This control type is generally used to store information between client/server exchanges that would otherwise be lost due to the stateless nature of HTTP. The input element is used to create a hidden control.

object control

You can insert generic objects in forms such that associated values are submitted along with other controls. Object controls are created with the object element.

 
     
  12.4 The form Element  
 

Syntax    <form>...</form>

List of form Attributes

The form element acts as a container for controls. It specifies:

  • The layout of the form (given by the contents of the form element).
  • The program that will handle the completed and submitted form (specified as the value of the action attribute). The receiving program must be able to parse name/value pairs in order to make use of them and are usually CGI programs.
  • The method by which user data will be sent to the server (specified as the value of the method attribute).

A form can contain text and markup (paragraphs, lists, etc.) in addition to form controls.

Below is the example given in Eg.1 above and shows that the form will be processed by the "userinfo.pl" CGI program when submitted. The form will be sent to the program using the HTTP "post" method.

 
<form name="Creditcard" action="http://www.somesite.com/cgi-bin/userinfo.pl" method="post">
Card Holders Name: <input type="text" name="CustomerName" size="20"><br>
Card Number: <input type="text" name="Number" size="20" maxlength="20"><br>
Expiry Date: <input type="text" name="Month" size="2" maxlength="2" value="MM">
/<input type="text" name="Year" size="2" maxlength="2" value="YY"><br>
<input type="submit" value="Submit"> <input type="reset">

</form>

  
     
  12.5 The input Element  
 

Syntax    <input>

List of input Attributes

The control type defined by the input element depends on the value of the type attribute:

text

Creates a single-line text input control.

password

Like the control type defined by "text", but the input text is rendered in a way so as to hide the characters, for example, a series of asterisks. This control type is often used for sensitive input such as passwords.

checkbox

Creates a checkbox.

radio

Creates a radio button.

submit

Creates a submit button.

image

Creates a submit button which is an image. The value of the src attribute specifies the URI of the image that will form the button. You should provide alternate text for the image via the alt attribute in case the visual user agent is unable to render images or is configured not to.

reset

Creates a reset button.

button

Creates a push button. User agents will probably use the value of the value attribute as the button's label.

hidden

Creates a hidden control.

file

Creates a file select control. User agents may use the value of the value attribute as the initial file name.

  12.7 The button Element  
 

Syntax    <button>...</button>

List of button Attributes

Buttons created with the button element function just like buttons created with the input element. However, the button element may have content and therefore they offer richer rendering possibilities. For example, a button element that contains an image functions like and may resemble an input element whose type is set to "image", but the button element type allows content. Visual user agents may render button buttons with relief and an up/down motion when clicked, while they may render input buttons as flat images.

Eg.4 The example below is similar to Eg.1 above, but creates the submit and reset buttons with the button element instead of the input element. The buttons contain images created by using the img element.

Expiry Date: <input type="text" name="Month" size="2" maxlength="2" value="MM">
/<input type="text" name="Year" size="2" maxlength="2" value="YY"><br>
<button name="submit" value="submit" type="submit"> 
SEND <img src="/images/submit.gif" width="50" height="30" alt="Click to submit form"></button>
<button name="reset" type="reset"> 
RESET <img src="/images/reset.gif" width="50" height="30" alt="Click to reset form"></button>
</form>

Do not use an image map with an img element that appears as the contents of a button element as this is not allowed.

 

  
     
  12.8 The select, optgroup, and option Elements  
 

Syntax    <select>...</select>

List of select Attributes

Syntax    <optgroup>...</optgroup>

List of optgroup Attributes

Syntax    <option>...</option>

List of option Attributes

The select element creates a menu. Each choice offered by the menu is represented by an option element. A select element must contain at least one option element.

The optgroup element allows you to group choices logically. This is useful when the user has to choose from a long list of options since groups of related choices are easier to work with than a single long list of options. Groups may not be nested, therefore all optgroup elements must be specified directly within a select element.

 

  
     
  12.9 Pre-selected Options  
  No choice, or one or more choices may be pre-selected for the user. User agents should determine which choices are pre-selected as follows:
  • If no option element has the selected attribute set, user agent behavior for choosing which option is initially selected is undefined. As user agent behavior differs, you should ensure that each menu includes a default pre-selected option element.
  • If one option element has the selected attribute set, it should be pre-selected.
  • If the select element has the multiple attribute set and more than one option element has the selected attribute set, they should all be pre-selected.
  • It is considered an error if more than one option element has the selected attribute set and the select element does not have the multiple attribute set.

When rendering a menu choice, user agents should use the value of the label attribute of the option element as the choice. If this attribute is not specified, user agents should use the contents of the option element.

The label attribute of the optgroup element specifies the label for a group of choices.

Eg.5 The following example creates a menu that allows the user to select which of six items he or she is, for example, interested in buying. The first, second and third special offer items are pre-selected but may be deselected by the user. The remaining items are not pre-selected. The size attribute states that the menu should only have 5 rows even though the user may select from among 6 options. The other options are made available through a scrolling mechanism.

The closing select tag is followed by the input elements and their attributes and values for forming the submit and reset buttons.

 
<form action="http://www.somesite.com/cgi-bin/order_processor.pl" method="post">
   <p>
   <select multiple size="5" name="selections">
      <option selected value="Special_Item_1">Special_Item_1</option>
      <option selected value="Special_Item_2">Special_Item_2</option>
      <option selected value="Special_Item_3">Special_Item_3</option>
      <option>Item_4</option>
      <option>Item_5</option>
      <option>Item_6</option>
   </select>
   </p>
   <p>Hold down the Ctrl key to make multiple selections.</p>
   <p><input type="submit" value="Send"> <input type="reset"></p>
</form>

Only selected options will be successful when using the select attribute/value pair of name="selections". When no options are selected, the control is not successful and neither the name nor any values are submitted to the server when the form is submitted. Note that where the value attribute is set, it determines the control's initial value, otherwise it's the element's contents.

Eg.6 In the following example, use is made of the optgroup element to group choices:

 
<form action="http://www.somesite.com/cgi-bin/program_name.pl" method="post">

  12.10 The textarea Element  
 

Syntax    <textarea>...</textarea>

List of textarea Attributes

The textarea element creates a multi-line text input control. User agents will use the contents of this element as the initial value of the control and will render this text initially.

Eg.7 The following example creates a textarea control that is 10 rows by 70 columns and contains one line of text initially. The textarea is followed by submit and reset buttons:

 
<form action="http://somesite.com/cgi-bin/somescript.pl" method="post">
   <p style="text-align:center">
   <textarea name="sometext" rows="10" cols="70">
   Type your comments in this text box.
   </textarea>
   </p>
   <p style="text-align:center"><input type="submit" value="Send"> <input type="reset"></p>
</form>

Setting the readonly attribute allows you to display unmodifiable text in a textarea. This differs from using standard marked-up text in a document because the value of textarea is submitted with the form. This may be useful in situations such as when a form has an agreement in it and the user has to acknowledge that he or she has read and abides by the agreement or conditions set forth before submitting the form.

 

  
     
  12.11 Labels  
  Some form controls (e.g. buttons) automatically have labels associated with them while most do not (e.g. text fields, radio buttons, checkboxes etc.). For those controls that have implicit labels, user agents will use the value of the value attribute as the label string. The label element is used to specify labels for controls that do not have implicit labels.  
     
  12.12 The label Element  
 

Syntax    <label>...</label>

List of label Attributes

To attach information to controls the label element may be used. Each label element is associated with exactly one form control.

The for attribute associates a label with another control. Also, the value of the for attribute must be the same as the value of the id attribute of the associated control element. More than one label may be associated with the same control by creating multiple references via the for attribute.

Eg.8 The following is an example:

    <label for="familyname">Family name: </label>
              <input type="text" id="familyname"><br>

  12.14 Disabled And Read-Only Controls  
  In situations where user input is unwanted, it is possible to disable a control or render it read-only. For example, you may want to include some read-only text that must be submitted as a value along with the form. This could be an agreement or set of conditions, for example.

Disabled controls

Disabled: When set for a form control, this attribute disables the control for user input.

The following elements support the disabled attribute: button, input, optgroup, option, select, and textarea.

When the disabled attribute is set it has the following effects on an element:

  • Disabled controls do not receive focus.
  • Disabled controls cannot be successful, i.e. their values will not be submitted with the form.

Eg.10 In the following example, the input element is disabled. Therefore, it cannot receive user input and its value will not be submitted with the form.

<input type="text" name="somename" value="somevalue" disabled>

Read-only controls

readonly: When set for a form control, this attribute prohibits changes to the control.

Using the readonly attribute prevents the control from being modified by the user.

The following elements support the readonly attribute: input and textarea.

When set, the readonly attribute has the following effects on an element:

  • Read-only elements receive focus but cannot be modified by the user.
  • Read-only elements are successful.

  12.15 Form Submission Methods  
  The method attribute of the form element specifies the HTTP method used to send the form to the processing agent. This attribute may have one of the following two values:
  • get: With the HTTP "get" method, the form data set is appended to the URI specified by the action attribute using a question-mark ("?") as separator and this new URI is sent to the processing agent.
  • post: With the HTTP "post" method, the form data set is included in the body of the form and sent to the processing agent.

The "get" method should be used when the form is idempotent, that is, causes no side-effects. Many database searches have no visible side-effects and make ideal applications for the "get" method.

If the service associated with the processing of a form causes side effects, for example, if the form modifies a database or subscription to a service, the "post" method should be used.

Usually the script on the server that is used as the processing agent will specify which of the attribute values to use (i.e. either get or post). Therefore, it is always advisable to check the script's documentation before setting this attribute's value in a form.

 

  
  12.16 Successful Controls  
  A successful control is one that is valid for submission. Every successful control has its control name paired with its current value as part of the submitted form data set. A successful control must be defined within a form element and must have a control name.

However:

  • Controls that are disabled cannot be successful.
  • All checkboxes that are "on" may be successful.
  • Only the "on" radio button may be successful for radio buttons that share the same value of the name attribute.
  • For menus, the control name is provided by a select element and values are provided by option elements. Only selected options may be successful. When no options are selected, the control is not successful and neither the name nor any values are submitted to the server when the form is submitted.
  • The current value of a file select is a list of one or more file names. Upon submission of the form, the contents of each file are submitted with the rest of the form data. The file contents are packaged according to the form's content type.
  • If a form contains more than one submit button, only the activated submit button is successful.

If a control doesn't have a current value when the form is submitted, user agents are not required to treat it as a successful control.

Hidden controls and controls that are not rendered because of style sheet settings may be successful.

 

  
  12.17 Drop-Down Navigation Menu  
  Eg.12 You can use the following example to create a drop-down navigation menu. These are often useful when you need to preserve space on your web pages. Place the following code in red between the head tags of your page (if you are embedding it in your web pages) or place it in your external script document (in which case you should leave out the opening and closing script tags). Then reference the external script document in your web pages.
 
<script language="JavaScript">
<!-- Hide from older browsers
function load(link) 
 {       
 var URL = link.options[link.selectedIndex].value 
 window.top.location.href = URL
 link.selectedIndex=0
 }
 // End Hide -->
</script>

Now place the following code in red between the body tags of your document at the place where you want the drop-down navigation menu to appear.

 
<form name="navigation">
<select name="navmenu" size="1" onChange="load(this)" style="color: #ffffff; 
font-family: verdana; font-weight: bold; font-size: 10px; background-color: #777777">
<option value="">Navigation Menu
<option value="index.htm">Homepage
<option value="products.htm">Our Products
<option value="prices.htm">Our Prices
<option value="specials.htm">Special Offers
<option value="newsletter.htm">Free Newsletter
</select>
</form>

This drop-down menu does not have a GO button, i.e. once a link is clicked on, it is immediately followed and the new web page starts to load.
